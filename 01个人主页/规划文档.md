# **第二大脑仓库结构审阅与建议**

## **仓库概览**
你将 Obsidian 笔记仓库公开到 GitHub，我们通过 GitHub API 浏览了主要目录。仓库的顶层包含以下结构：

- **01个人主页**：包含一个 主页.md 文件（为空）以及一个 .canvas 画布文件。
    
- **02root**：根目录下有 Inbox(收集箱) 和 Project(项目) 两个子目录 。
    
- **03daily**：存放每日笔记，每条笔记以日期命名，内容基于同一模板，包括 GTD & Top3 任务、Morning checklist、复盘和每日习惯跟踪等 。
    
- **04work**：工作区，内部有 agent prompt、周会、工作项目、面试 等子目录 。
    
- **Excalidraw**：用于存放 Excalidraw 画布。
    
- **template**：笔记模板集合，包含日记模板、Callouts 模板以及“新任务”、“新项目”等标签模板；日记模板与实际的每日笔记内容几乎一致 。
    
- 其他零散文件包括 HTML代码块、obsidianURL联用.md（讲解 Obsidian URL Scheme ）、idea.md（记录想法）以及 参考文档.json 等。
    

  

## **结构优点**

- **使用模板规范笔记格式**：日记模板提供了统一的结构，包括待办、复盘、Dataview 查询和习惯检查等 。
    
- **分类初具雏形**：你将收集箱、项目、每日笔记、工作笔记分开存放，避免所有内容堆在根目录。
    
- **前置 metadata**：大部分笔记在开头使用 YAML 前置数据记录类型、日期、标签等，便于 Dataview 查询和后续自动化管理。
    
- **存放模板文件的专用目录**：将模板放在 template 目录，避免和正式笔记混淆。
    

  

## **改进建议**

  

### **1 顶层结构建议：采用 PARA/PPAC 框架**

  

当前根目录只有 Inbox 和 Project 两个子目录 。可以参考 PARA（Projects、Areas、Resources、Archives）或 PPAC（Projects、People、Areas、Content）等方法，重新规划顶层结构：

- **Inbox / 收集箱**：用来临时捕捉想法、链接等；定期清理并将内容归档到其他类别。现在的收集箱中混杂了 Mer‑maid 教程与 Obsidian URL 文章 ，应在学习后移动到合适的“Resources”或“Guides”目录，并删除临时文件。可以设立 01 Inbox 作为单独目录，避免放在根目录里难以发现。
    
- **Projects (项目)**：用于放置有时间期限和明确目标的项目资料，例如 opencompass 或 “复杂指令遵循”等 。建议为每个项目建立子目录，包含项目简介、待办事项、会议记录、决策日志等。当项目结束后，移动到档案目录归档。
    
- **Areas (领域/责任)**：长期关注、需要持续维护的领域，如职业成长、个人健康、财务管理、学习计划等。当前仓库没有此目录，可根据生活/工作领域创建 Area(领域) 目录，并在其中维护相关笔记。例如可以把 工作项目 下的 opencompass 放到 Area：职业项目；把 面试 相关笔记放在 “职业发展” 领域。
    
- **Resources (资料/知识库)**：存放不属于特定项目的文章、教程、方法论等。例如 mermaid 与 Obsidian URL 笔记都更像是知识参考，应放入 Resources 而非收集箱 。可以按主题再划分二级目录（如 “工具教程”、“学习笔记”、“引用文档”），并利用 YAML 前置资料存储作者、来源链接等信息。
    
- **Archives (归档)**：保存已完成的项目笔记或不再经常访问的材料，以保持主库简洁。
    

  

### **2 个人主页 (01个人主页)**

  

01个人主页 中的 主页.md 文件为空，而 .canvas 文件无法在纯文本浏览器直接查看。建议：

- **作为门面与导航页**：主页可以设置为“第二大脑仪表盘”，包含个人简介、当前焦点、核心价值观、年度/季度目标以及指向其他目录的链接。例如添加到 主页.md 中的内容包括：
    
    - 个人愿景与使命宣言。
        
    - 当前活跃项目（可使用 Obsidian Dataview 从 Projects 自动列出）。
        
    - 常用链接（例如知识库、任务列表、习惯追踪）。
        
    
- **使用 Canvas 可视化关系**：现有的 .canvas 文件可用来绘制知识库关系图或年度计划。可将 Canvas 与 主页.md 互相链接，方便在不同视图间切换。
    

  

### **3 收集箱 (Inbox)**

- **仅用来快速捕获**：收集箱中的笔记代表“尚未分类处理”的信息。如 mermaid.md 与 url.md 两篇文章都已经完整写成教程，说明你在收集箱里完成了整理 。建议每周查看收集箱，将笔记移动到合适的目录（比如 Resources），并在收集箱留下提醒或删除原笔记，确保收集箱始终保持清爽。
    
- **分层次捕获**：可以在收集箱下创建 Ideas、Links、Notes 等子目录，用于临时区分材料；但不要囤积太久，避免收集箱变成永久存储。
    

  

### **4 Projects (项目)**

- **明确项目目录**：02root/Project(项目) 目前为空 。项目是有明确目标和截止日期的任务集合，可以将工作和生活中的项目统一归档到此目录。例如 04work/工作项目 下的 opencompass、复杂指令遵循、系统指令遵循 等都属于项目，建议移动到 Projects 并按项目名称创建子目录 。
    
- **标准化项目笔记**：每个项目目录建议包含：
    
    - 项目简介 (README.md)：说明项目目标、成功标准、参与人等；
        
    - 任务列表 (tasks.md)：记录待办及进度，使用模板或 dataview；
        
    - 会议记录 (meetings 文件夹)；
        
    - 决策记录 (decisions.md)，总结关键决策和原因。
        
    
- **项目归档**：项目完成后，可移动到 Archives 并更新主页仪表盘。
    

  

### **5 Daily (03daily) 与模板**

- **模板与实例区分**：每日笔记模板和实际笔记都使用了同一结构 。模板放在 template 目录中，实际日记放在 03daily。模板与实例之间应通过 Obsidian Templater 插件调用。建议在模板中添加更多变量（如日期、天气、情绪），并移除冗余字段。例如模板 front‑matter 中的 Ai_learning、English_learning 等复选框可根据你的具体习惯定制。
    
- **合理设置 front‑matter**：每日笔记的 front‑matter 中字段过多会增加维护负担。可以考虑简化为 type: daily、date: [[<% tp.date.now("YYYY-MM-DD") %>]]等必需字段，同时用 Dataview 视图在主页聚合打卡数据。
    
- **周/月回顾**：在 03daily 外建立 weekly 和 monthly 目录，使用模板引导回顾；或通过 Dataview 在个人主页展示过去一周完成的任务和习惯统计。
    

  

### **6 工作区 (04work)**

- **重构目录结构**：当前 04work 目录下混合了会议记录、项目、面试笔记以及 prompt 文件 。建议将所有工作相关内容纳入顶层 Projects / Areas 框架，04work 可重命名为 Work 或拆分为：
    
    - Work/Meetings：存放周会、1:1 等会议记录；每条会议笔记设置 type: meeting、date 等 front‑matter，使用模板规范出席者、议题和行动项 。
        
    - Work/Prompts：存放如 agent prompt/prompt.md 这类 AI 提示语，以免混淆 。
        
    - Work/Interview：将面试笔记集中存放，比如 2025‑06‑25‑面试.md；面试笔记可以包含候选人信息、问题、评价等，独立于项目 。
        
    - Work/Projects：工作项目子目录，与 Projects 同级或合并。
        
    
- **明确笔记类型**：给每条工作笔记设置 type: meeting、type: interview 或 type: prompt 等，便于 Dataview 筛选。
    
- **会议笔记模板**：为周会等创建 meeting 模板，包括参会人、议程、讨论内容、决定和行动项，像你现有的周会笔记中所记录的那样 。
    

  

### **7 模板管理 (template)**

- **模板命名统一**：模板文件夹的中文命名可能会导致路径和命令混乱。建议统一使用简洁的英文或数字前缀，例如 Templates/Daily.md、Templates/Project.md、Templates/Task.md。
    
- **充分利用 Templater 功能**：现有 New-Task.md 模板为任务设置了状态、进度、截止日期、优先级等变量 。可以根据自己的 workflow 添加更多字段，如标签、预算、预计工作量等。
    

  

### **8 其他建议**

1. **建立 index / MOC（地图）笔记**：为每个重要目录创建一个地图笔记，列出该目录下的子笔记和关键链接，便于快速导航。例如为 Resources 创建 Resources MOC.md，列出各类教程和参考文档。
    
2. **规范文件命名**：避免在文件名中使用空格和复杂字符，使用统一的日期格式（如 YYYY-MM-DD.md），确保排序一致。对于中文名称的目录或文件，可以在前面加上英文前缀以便于引用。
    
3. **使用标签和 YAML 元数据**：通过标签表示笔记性质（如 #meeting、#idea），再结合 YAML front‑matter 字段（type、date、project 等），便于 Dataview 和任务管理插件进行过滤和统计。
    
4. **及时整理**：收集箱、Inbox 和 Ideas 目录需要定期清理，避免长期堆积导致信息过载。
    
5. **参考实践**：阅读和学习 PARA、GTD、Zettelkasten 等知识管理方法，根据个人需求灵活调整，不必拘泥于单一框架。
    

  

## **总结**

  

你的 Obsidian 仓库已经具备较为完整的模板和基本分类，初步实现了第二大脑的构建。下一步应着重

- 调整目录结构，按照 PARA/PPAC 框架明确项目、领域和资源的归属；
    
- 填充个人主页，建立仪表盘与 MOC；
    
- 优化 Inbox 的使用，并及时归档收集的信息；

- 通过标签和 Dataview 制作看板和统计，提升笔记的可查询性。
    

  

坚持使用并不断优化结构，你的第二大脑将更加高效、清晰，能够助力学习、工作与个人成长。
